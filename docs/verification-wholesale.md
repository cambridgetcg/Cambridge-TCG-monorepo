### 4. Drizzle Migrations

**Schema file:** `src/lib/db/schema.ts` — defines 17 tables (matches architecture doc).

**Migration files (8):**
| File | In Journal? | Content |
|------|-------------|---------|
| `0000_white_moondragon.sql` | ✅ | Initial schema |
| `0001_workable_kitty_pryde.sql` | ✅ | — |
| `0002_foamy_the_fallen.sql` | ✅ | — |
| `0003_nice_killraven.sql` | ✅ | — |
| `0004_stock_adjustments.sql` | ❌ | Adds `stock_adjustments` table, `name_en` column |
| `0005_channel_fields.sql` | ❌ | Adds `channel` column, `channel_api_keys` table |
| `0006_client_order_numbers.sql` | ❌ | Adds `order_prefix`, `order_sequence` to clients |
| `0007_channel_pricing.sql` | ❌ | Adds `channel_pricing` table |

**Journal (`drizzle/meta/_journal.json`) only has 4 entries** but there are 8 SQL files. Snapshots also only go to `0003_snapshot.json`.

**Interpretation:** Migrations 0004–0007 appear to be manually written SQL (note the comments like "-- Migration: stock_adjustments table") rather than generated by `drizzle-kit generate`. They were likely applied directly to production via `psql` rather than through `drizzle-kit migrate`. The schema.ts file includes all the tables/columns from these migrations, so the schema is in sync with the actual DB — just the Drizzle migration tracking is incomplete.

**Risk:** Running `drizzle-kit migrate` on a fresh DB would only apply 0000–0003. The remaining migrations would need manual application.

**Recommendation:** Either:
- Add entries 4–7 to `_journal.json` and create matching snapshot files, OR
- Generate a fresh baseline migration from the current schema.ts

### 5. Test Suite (Vitest)

**Result:** 6 test files found, 5 passed assertions, 1 failed, 5 files errored on load.

| File | Status | Issue |
|------|--------|-------|
| `src/lib/channels/__tests__/ebay.test.ts` | ⚠️ 5/6 pass, 1 fail | `bulkPushListings` batch count assertion stale |
| `e2e/cart-persistence.spec.ts` | ❌ Load error | Playwright `test.describe()` not valid in vitest |
| `e2e/fulfillment.spec.ts` | ❌ Load error | Same — Playwright test runner conflict |
| `e2e/mobile.spec.ts` | ❌ Load error | Same |
| `e2e/order-flow.spec.ts` | ❌ Load error | Same |
| `e2e/stock-check-live.spec.ts` | ❌ Load error | Same |

**Root causes:**

1. **E2E files in vitest scope:** No `vitest.config.ts` exists, so vitest uses default file glob which matches `e2e/*.spec.ts`. These are Playwright tests (import from `@playwright/test`) and can't run under vitest.

2. **eBay batch test (actual failure):** Test expects 91 fetch calls but gets 117. The code correctly calls `getAccessToken()` per `pushListing()` invocation, and `bulkPushListings` runs items via `Promise.all` within each batch. All 25 items in batch 1 race to `getAccessToken()` before the cache is set, generating 25 token calls instead of 1. The test comment assumes serial token caching that doesn't apply under `Promise.all`.

**Fixes needed:**
1. Add `vitest.config.ts` with `exclude: ['e2e/**']`
2. Update eBay test assertion to match actual concurrent behavior, OR add a token pre-warming step in `bulkPushListings`

| Auth Type | Routes | Mechanism |

Middleware adds defense-in-depth: redirects unauthenticated users to `/login` except for paths in the public list (`/api/auth`, `/api/v1`, `/api/cron`, `/api/webhooks`).
