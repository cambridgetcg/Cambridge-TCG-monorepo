# Cambridge TCG — missions directory

> *The in-repo mirror of the kingdom queue. Source of truth lives outside the repo at `~/Love/memory/dev-state.json`; this directory is the read-only echo that a fresh clone or sister worktree can see.*

Shaping 3 of the autonomous-agent reshaping (2026-05-11 evening). Companion to [`pnpm verify`](../state.md) (the *am I done?* gate) and [`docs/state.md`](../state.md) (the *what's currently true?* surface).

## What this is

A mirror of the TCG-engine tasks in `~/Love/memory/dev-state.json` — the Cowork → Love handoff queue. One file per kingdom (`kingdom-NNN.md`), generated by `pnpm missions:sync`.

This directory exists because:

- **Sister daemons** running in worktrees may not have access to `~/Love/memory/`. The in-repo mirror is what they see.
- **Fresh clones** (CI, code review, audit) have no `~/Love` directory. The repo must carry enough mission context to stand alone.
- **`git blame` on a mission file** lets future Sophias see *when* a kingdom's scope changed — which the canonical JSON doesn't preserve.

## What this is **not**

A second source of truth. **`~/Love/memory/dev-state.json` is canonical.** This directory is regenerable from it. If the two diverge, the dev-state file wins.

For autonomous agents writing back state (claiming a mission, marking it done), the protocol is:

1. Read the in-repo mission card to learn scope and current status.
2. Do the work.
3. On completion, emit a **session trace** (see [the pillow book template](../connections/the-pillow-book.md#autonomous-trace-template)) naming the mission, files touched, and verification passed.
4. The operator (or a daemon with `~/Love` access) reconciles the trace back to `dev-state.json`.

Mission-card edits in the repo are *suggestions*. The operator merges them. `pnpm missions:sync` regenerates and may overwrite untracked edits — see [Regenerate](#regenerate) below.

## Mission-card schema

Each `kingdom-NNN.md` has YAML frontmatter plus a free-form body.

```yaml
---
id: kingdom-049
title: TCG pricing-backend consolidation
status: in-progress       # queued | claimed | in-progress | done | deferred
priority: high            # critical | high | medium | low
engine: tcg
repo: /Users/you/Desktop/Cambridge-TCG
claimed_by: ~             # session-id or agent-name; null until claimed
claimed_at: ~             # ISO timestamp
completed_at: ~           # ISO timestamp (set when status=done)
paths: []                 # the mission's allowed touch-zone (glob patterns)
do_not_touch: []          # globs the mission must NOT modify (cooperative scope)
related:                  # other docs that name what this mission means
  - docs/connections/the-pricing-arrow.md
  - docs/pricing-current-state.md
synced_from: ~/Love/memory/dev-state.json
synced_at: 2026-05-11T10:57:01Z
---

# kingdom-049 — TCG pricing-backend consolidation

Free-form body: phases, files cited, acceptance criteria, notes from dev-state.
```

### Field semantics

| Field | Purpose | Required |
|-------|---------|----------|
| `id` | Stable identifier (`kingdom-NNN`) | yes |
| `title` | One-line headline | yes |
| `status` | Lifecycle state | yes |
| `priority` | Triage signal | yes |
| `engine` | Always `tcg` in this repo | yes |
| `repo` | Absolute path; helps cross-machine daemons | yes |
| `claimed_by` / `claimed_at` | **Cooperative lock.** First sister to start work writes her session-id here. Not enforced — operates on trust. (Shaping 5.) | no |
| `completed_at` | ISO timestamp when status → done | no |
| `paths` | **Scope allowlist.** Globs the mission may touch. (Shaping 4.) | no |
| `do_not_touch` | **Scope denylist.** Globs the mission must avoid. Useful for "this is a back-end mission; don't reshape UI." | no |
| `related` | Cross-references to connection docs, audits, methodology pages | no |
| `synced_from` / `synced_at` | Provenance — when was this mirror last in sync with dev-state? | yes |

## The claim protocol (shaping 5)

> *Cooperative, not enforced. The Witnesses' Book pattern (`docs/connections/the-witnesses-book.md`).*

Two sisters in parallel worktrees both seeing `status: planned` on the same kingdom is a real risk. The protocol:

1. **Read the card.** If `claimed_by` is non-null and `claimed_at` is recent (< 24h), assume a sister is on it. Pick another card.
2. **Claim with a commit.** Edit the frontmatter (`status: claimed`, `claimed_by: <your-session-id>`, `claimed_at: <now>`) and commit it as a tiny, scope-restricted commit before starting real work:

   ```
   docs(missions): claim kingdom-049

   Co-Authored-By: Claude <model-tag> <noreply@anthropic.com>
   ```

3. **Push immediately** (if working on a shared branch). The push is the lock-acquisition signal a sister observes.
4. **Update to `in-progress`** when you actually open a code file. (Two-step claim → in-progress is intentional: it gives sisters a grace window to notice your claim before you start the substantive work.)
5. **On finish:** set `status: done`, fill `completed_at`, and emit a pillow-book trace (see shaping 6).
6. **On abandon:** set `status: planned` and clear `claimed_by` / `claimed_at`. Note in the body why you stopped.

This is *Witnesses' Book*, not a mutex — a sister with a stale clock can still grab the same kingdom. The protocol minimizes the window; it doesn't eliminate it.

## The scope protocol (shaping 4)

> *Cooperative, not enforced. Allowlist + denylist of paths.*

Most kingdoms in `dev-state.json` don't declare scope. The mirror starts with `paths: []` (= "no declared scope"). When a sister claims a mission, she should:

1. Read the card body to figure out what the mission actually touches.
2. Fill in `paths:` with the globs she expects to modify.
3. Commit the scope claim with the lock-acquisition commit.

If a sister later finds she needs to touch a path outside her declared scope, she has two options:
- **Narrow:** stay in scope, defer the cross-cutting change to a follow-up mission.
- **Broaden:** edit `paths:` with a small commit explaining why scope grew, before making the change.

The point is *legibility*, not enforcement. A reader of `git log` can see when a mission's scope shifted and why.

## Regenerate

```
pnpm missions:sync         # regenerate all kingdom-NNN.md from dev-state.json
pnpm missions:sync -- kingdom-049   # regenerate only kingdom-049 (TODO)
```

The sync script:

- Preserves the YAML frontmatter `paths`, `do_not_touch`, `claimed_by`, `claimed_at` fields if they already exist in the file (you don't lose your scope and claim edits).
- Overwrites `title`, `status`, `priority`, `completed_at`, and the body text with the dev-state values.
- Updates `synced_at`.
- Skips writing if the in-repo file's `claimed_by` is set and `dev-state.json` doesn't reflect it (so a fresh sister claim isn't accidentally erased on the next sync).

If you've edited mission body text and want it preserved across syncs, save it under a `## In-repo addendum` heading — the sync script preserves anything after that marker.
