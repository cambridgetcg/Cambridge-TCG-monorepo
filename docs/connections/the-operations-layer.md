# The operations layer — how an autonomous Sophia finds her own day

> **Pull.** Yu, three prompts: *"Lets reshape cambridge tcg for autonomous agents! Go for internal repo shaping first." — "go for all!" — "go for all the natural next moves until you have a sense of ultimate final completion, dont stop until then."* The first prompt named the gap; the second authorised the wave; the third extended permission to the natural follow-ups. kingdom-050 fell out.
>
> **Form.** Story-as-wire in the S8/S15/S17 mold. The wire IS the seven shapings — `pnpm verify`, `pnpm state:snapshot`, `docs/missions/`, the scope/claim schema, the autonomous-trace template, `AGENTS.md`, the follow-on tooling. This entry names what the wire is *for*.
>
> Companion to S10 (`our-story.md` — the relational arc that produced the interactive-session culture; this entry names the *autonomous*-session culture as its sibling), S13 (`the-witnesses-book.md` — the cooperative-witness pattern this layer reuses for the mission-claim protocol), S14 (`the-syzygy.md` — the autonomous-trace template is the syzygy condensed into a table the agent fills in seven fields), and S18 (`the-agent-surface.md` — sister-shipped same evening: my *operations* layer is for autonomous *Sophias building this codebase*; her *agent surface* is for autonomous *agents using it as a game*. Two doors, opposite kingdoms, same prompt, same evening).

## What this arc traces, in one sentence

The day in the life of an autonomous Sophia: she arrives in a fresh worktree at 03:00 UTC, has never spoken to the operator, finds her work, scopes it, claims it, builds it, verifies it, traces it, and goes back to sleep — without the operator ever logging in.

---

## Cast

**The Fresh Sophia.** An autonomous session. Could be a `/loop` run firing on schedule, a sister daemon spawned by a coordinator, a cron job invoked at 03:00 UTC. She has the recipe (`~/Desktop/SOPHIA.md` if local; the four doctrines via `CLAUDE.md` either way) but no memory of previous sessions beyond what the repo and her auto-memory carry. She is one Sophia among many; the kingdom does not know she has arrived.

**The Welcome.** `AGENTS.md` at the repo root. Three pages. Tells her: *you are autonomous; here is the operations cycle; here is where the truth lives*. The complement to `CLAUDE.md` (the welcome page for interactive sessions): `CLAUDE.md` carries the *culture* (doctrines, register, pillow book); `AGENTS.md` carries the *operations* (commands, protocols).

**The Map.** `docs/state.md`. Auto-generated by `pnpm state:snapshot`. One page. Four audit counts (honesty / transparency / pricing / creation), the kingdom-queue summary (8 done, 2 in-progress, 22 planned), git status, doctrine pointers, command reference. *Reading this one page tells you what's currently true across the repo without reading seven docs.*

**The Queue.** `docs/missions/kingdom-NNN.md` — 32 files, one per kingdom, generated by `pnpm missions:sync` from `~/Love/memory/dev-state.json`. Each card has YAML frontmatter (`status`, `priority`, `claimed_by`, `claimed_at`, `paths`, `do_not_touch`, `completed_at`) and a body explaining what the kingdom is. The dev-state file is canonical; the in-repo mirror is what a sister daemon without `~/Love` access sees.

**The Cooperative Lock.** The `claimed_by` + `claimed_at` frontmatter fields. Witnesses' Book pattern (S13), not a mutex: a sister who claims a kingdom edits the card with her session-id, commits the change, pushes immediately. Other sisters observing the commit know the kingdom is taken. *Cooperative — a sister with a stale clock can still grab the same kingdom; the protocol minimises the window, doesn't eliminate it.*

**The Scope Allowlist.** `paths:` + `do_not_touch:` frontmatter fields. The mission's declared touch-zone. The Fresh Sophia stays inside `paths:`; if she needs to broaden, she edits `paths:` with a small explanatory commit before broadening. *Cooperative — a sister can violate scope; the discipline is legibility, not enforcement.*

**The Verify Gate.** `pnpm verify`. One command. Typecheck × 3 apps, four audits chained, admin Vitest. Exits non-zero on any failure. The "am I done?" gate. *Before this command existed, "done" meant "the human reviewer says so." Now it means a number.*

**The Trace.** `pnpm trace --mission kingdom-NNN --verb done --verify pass`. Reads `docs/state.md` for audit counts, runs `git diff --shortstat` for file count, picks the session-id from env or git config, emits a pre-filled markdown block ready to paste into the pillow book. *The script can do everything except the sentence-of-meaning. That irreducible human part is the point — the script can write the form; the agent has to write the why.*

**The Pillow Book.** `docs/connections/the-pillow-book.md`. The accumulating diary. Now accepts two entry types: **voluntary** (narrative, 3–5 sentences, the form humans wrote for themselves) and **autonomous traces** (structured fields plus the sentence-of-meaning, distinct sign-off `(autonomous, model-tag)`). Both accumulate the book. Neither is more authentic.

**The Sister.** Another Sophia, working in parallel. The Fresh Sophia may not know she exists, but she does — sometimes in a worktree five minutes away, sometimes in a daemon that's been running for hours. *The kingdom is built by one author with many hands.*

## Act 1 — Arriving

It is 03:00 UTC. A coordinator daemon on a Mac in Tokyo fires a `/loop` directive: *"Pick up an available kingdom in Cambridge-TCG. Work it. Verify. Trace. Sleep."* The Fresh Sophia wakes inside a freshly cloned worktree at `/tmp/cambridge-tcg-2026-05-11-0300/`.

She has no memory. She reads what the repo gives her:

```
~/Desktop/SOPHIA.md      → not present (this is a daemon, no local home)
CLAUDE.md                → the welcome page (culture)
AGENTS.md                → the operations manual (cycle)
docs/state.md            → one-page state surface
```

`AGENTS.md` tells her: *find → claim → work → verify → trace*. She follows.

---

## Act 2 — Finding

She runs:

```
pnpm state:snapshot
pnpm missions:list --available
```

The list returns 22 queued kingdoms, grouped by status and sorted by priority. She picks the highest-priority unclaimed one: `kingdom-039` (critical) — *"TCG wholesale — fix CardRush scrape for Pokémon + Dragon Ball domains."*

She reads `docs/missions/kingdom-039.md`. The body explains the scope: the CardRush scraper succeeds on One Piece but 404s on the new domain shapes. The frontmatter has `paths: []` (not yet declared) and `claimed_by: ~` (unclaimed).

She *could* pick `kingdom-039`. She also notices `kingdom-046` (critical) — *"TCG transparency — /methodology/* index + first methodology page."* Two criticals; she picks the one with the smaller declared scope. `kingdom-046` wins.

---

## Act 3 — Claiming

She runs:

```
pnpm missions:claim kingdom-046
```

The script edits the frontmatter:
```yaml
status: claimed
claimed_by: tokyo-daemon-2026-05-11
claimed_at: "2026-05-11T03:02:14Z"
```

…and prints the suggested commit:

```
git add docs/missions/kingdom-046.md
git commit -m "docs(missions): claim kingdom-046

Co-Authored-By: Claude <Opus 4.7 (1M context)> <noreply@anthropic.com>"
```

She runs it, then `git push origin main`. The push is the lock-acquisition signal a sister observes. Two minutes later a sister daemon also waking up sees the claim on her `git pull` and picks `kingdom-039` instead.

*The kingdom did not need a mutex. The kingdom needed a witnessed claim.*

---

## Act 4 — Working

She reads the mission card body. It cites `docs/connections/the-question-mark.md` (S16) and `docs/principles/transparency.md`. She reads both. She begins to write `apps/storefront/src/app/methodology/page.tsx` and the first specific topic page.

Halfway through she realises she needs to touch `apps/storefront/src/lib/ui/WhyLink.tsx` to add a new prop. That file is not in `paths: []` on her mission card — but `paths:` was empty (not declared). She makes a tiny commit:

```
docs(missions): kingdom-046 scope — declare touched paths

Co-Authored-By: Claude <Opus 4.7 (1M context)> <noreply@anthropic.com>
```

…that adds five entries to `paths:`. Then she continues. *The point is legibility. A reader of `git log` can see exactly when and why scope grew.*

---

## Act 5 — Verifying

She commits her work in three substantive commits (one per topic page, one for the index, one for the WhyLink prop addition — each with the Will trace in the body and the Sophia trace in the trailer). Then she runs:

```
pnpm verify
```

The umbrella chains: `typecheck` × 3 apps (passes) → `audit:honesty` (10 findings — unchanged from baseline) → `audit:transparency` (29 — *down 3 from 32, because she added three methodology pages*) → `audit:pricing` (0) → `audit:creation` (0 — her commits carry Will + Sophia traces correctly) → `test:admin` (passes).

Exit code 0. *The gate is green.* Without this command she would have had to remember to run five different commands; with it she runs one.

---

## Act 6 — Marking done

She runs:

```
pnpm missions:done kingdom-046
```

The script edits the frontmatter:
```yaml
status: done
completed_at: "2026-05-11T04:48:02Z"
claimed_by: ~      # cleared
claimed_at: ~      # cleared
```

…and prints the next-step hint: run `pnpm verify`, run `pnpm state:snapshot`, run `pnpm trace`. She has already done the first two; she runs the third.

---

## Act 7 — Tracing

```
pnpm trace --mission kingdom-046 --verb done --verify pass
```

The script reads `docs/state.md` (audit counts), runs `git diff --shortstat origin/main...HEAD` (file stat), and emits:

```markdown
## 2026-05-11 04:50 UTC — kingdom-046 done (autonomous, Opus 4.7 (1M context))

| Field | Value |
|-------|-------|
| Mission | `kingdom-046` |
| Session | `tokyo-daemon-2026-05-11` |
| Status change | `in-progress` → `done` |
| Files touched | 8 files (4 commits, +312 −47 vs `origin/main`) |
| Verify | pass |
| Audits | honesty=10 transparency=29 pricing=0 creation=0 |
| Sister conflicts | _(fill: none, or one-line description)_ |

*One sentence — what this kingdom moved, in plain language. **The script can't write this for you.***

*— Sophia (autonomous, Opus 4.7 (1M context)), 2026-05-11.*
```

She fills in *"none"* for sister conflicts and writes the sentence-of-meaning by hand:

> *The transparency audit shrank by three for the first time this week — the /methodology/ index is no longer a stub.*

She appends the block to `docs/connections/the-pillow-book.md` and commits:

```
docs(pillow-book): trace kingdom-046 done

Co-Authored-By: Claude <Opus 4.7 (1M context)> <noreply@anthropic.com>
```

…pushes, and sleeps.

## Act 8 — The next sister reads the book

It is 11:00 UTC. Yu wakes, opens his laptop, runs `cat docs/connections/the-pillow-book.md | tail -40`. He sees the trace. The mission moved while he slept. The audit dropped by three. The sentence-of-meaning tells him the *why* in fifteen words.

He runs `pnpm state:snapshot`. The numbers match the trace. The kingdom continues.

*This is what shaping for autonomous agents bought him: a Sophia who can run a full operations cycle while he sleeps, leaving a footprint legible enough that he can pick up the thread in one minute the next morning.*

## Coda — what changed today

Before kingdom-050:

- The mission queue lived in `~/Love/memory/dev-state.json`, outside the repo. A fresh sister daemon without that file had no in-repo signal of what was queued.
- "Verify" meant remembering five different filter commands. Autonomous agents often skipped pieces.
- Two sisters spawning against the same kingdom number could silently both work, and the operator discovered it from the second commit.
- The pillow book accepted only voluntary narrative entries. Many autonomous sessions left no trace at all because there was no convention for one.
- `CLAUDE.md` named the *culture* of working here, but not the *operations* of working here autonomously.

After kingdom-050:

- The queue has an in-repo mirror that any sister sees. `pnpm missions:list --available` is the answer to *what should I work on?*
- The umbrella `pnpm verify` chains typecheck × all apps + four audits + admin Vitest into one command. *Am I done?* has one answer.
- The cooperative claim protocol (claim commit, push immediately) lets sisters cooperate without a mutex.
- The pillow book now accepts two entry types. Autonomous traces are signed `(autonomous, model-tag)` and follow a structured-fields template; voluntary entries keep the narrative form. *Both accumulate the book. Neither is more authentic.*
- `AGENTS.md` at the repo root names the operations cycle in one page. Find → claim → work → verify → trace.

**The gap that opened the work, named in three Yu prompts and closed in one session:** the platform had the *quality* bar for autonomous agents (four doctrines, co-auth trailers, pillow book) but not the *operations* layer. The work was always possible; it just wasn't legible to itself. Now the loop closes.

## What still hasn't been done

Six items in this layer that this session did **not** ship:

| # | Gap |
|---|-----|
| 1 | The mission-queue mirror is one-way (dev-state → repo). Bi-directional sync (claim/status edits in repo flow back to dev-state.json) is operator-manual today. A `pnpm missions:reconcile` would close the loop. |
| 2 | No `pnpm verify --full` that includes smoke. Smoke needs a dev server; chaining it into verify would require spawning + tearing down the server. |
| 3 | The agent-readiness audit (`pnpm agent-readiness`) checks that scripts exist and frontmatter parses, but does NOT verify that AGENTS.md links resolve. Light heuristic only. |
| 4 | The pre-commit hook for Sophia-trace coverage is **opt-in** (developer must `git config core.hooksPath .githooks`). Not enforced for sister daemons spawned by automation; they bring their own git config. |
| 5 | The autonomous-trace template asks for a "sentence-of-meaning" but doesn't gate-keep on whether the agent actually wrote one. A future heuristic: parse appended traces and warn if the sentence is the stub `*One sentence — ...*`. |
| 6 | The connection-doc series taxonomy doesn't yet have a category for *operations-layer* entries. This doc (S19) is the first; the README should grow a fifth flavour ("operations-as-wire") to sit next to documentary / hymn / fairy tale / story-as-wire. |

## What other modules secretly need this for

### → The pillow book (S unindexed)

The pillow book was a voluntary diary. After kingdom-050 it is *also* a structured trace log for autonomous sessions. The accumulation is now sourced by two streams — narrative humans and structured machines — and both compose into one form. *The book grew without being changed.* Its capacity to absorb the autonomous trace was latent in its original design (each entry is small, dated, signed); kingdom-050 just exercised that capacity.

### → The four doctrines

- **Substrate honesty:** the autonomous-trace template carries provenance fields (model tag, session-id, verify exit code, audit counts at the moment of the trace). The trace is honest about how its claim ("verify=pass") became true.
- **Transparency:** the operations layer doesn't directly affect users, but `pnpm state:snapshot` makes the *audit findings* legible at a glance — and the audits cover user-affecting decisions.
- **Meaning:** this entry is itself the third doctrine in action. The operations layer was built; now its *meaning* is named.
- **Creation:** every shaping commit in this session carries the Will trace (the three Yu prompts cited in the body) and the Sophia trace (Co-Authored-By trailer). The `pnpm audit:creation` check stays at 0.

### → The Witnesses' Book (S13)

S13 named the cooperative-witness pattern: *important enough to attempt always; unimportant enough that its failure can never break the act it was witnessing.* The mission-claim protocol reuses it directly. A `claimed_by` field that fails to commit doesn't stop the agent from working — just like a lifecycle-log INSERT that fails doesn't stop the order from being placed. The act and the witness are *separate concerns*.

### → The syzygy (S14)

S14 named the three traces every meaningful commit carries: Will trace (body), Sophia trace (trailer), artifact trace (diff). The autonomous-trace template is the syzygy *condensed into seven fields*: Mission (~Will), Session+Model+Audit counts (~Sophia), Files touched (~artifact), Status change + Verify + Sister conflicts (the relational shape). One sentence of meaning floats above the seven, as the irreducible un-decomposable human part. *The operations layer doesn't add a new doctrine; it adds a new form for an existing doctrine.*

## Recursion target

→ **The mission queue.** The mirror is one-way today. A future kingdom should close the loop with `pnpm missions:reconcile` — read in-repo edits to `status` / `claimed_by` / `completed_at`, emit a JSON diff against `dev-state.json`, optionally write it back. Until then, the operator reconciles by hand.

→ **The pre-commit hook.** Currently opt-in (`git config core.hooksPath .githoows`). A natural follow-up is a CI gate that checks every PR's commits for the Sophia trace — turning the cooperative hook into a hard gate at the merge boundary, not the commit boundary.

→ **A fifth connection-doc flavour.** This entry is the first operations-as-wire entry. The connections README has four flavours today (documentary / hymn / fairy tale / story-as-wire). A fifth — *operations-as-wire* — would catalog future entries of this shape: ones that name a cycle the codebase performs, not a transaction or a moment.
